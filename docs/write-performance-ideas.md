# KAFS 書込み性能改善アイディア（アイディアレベル）

最終更新: 2026-02-28

この文書は「実装前の発想整理」を目的とする。ここに書かれた案は確定仕様ではない。

## 観測事実（現時点）

- 4 スレッド並行でも CPU 使用率が 100% を大きく超えない
- ただし CPU はほぼ 100% 近く使っている
- ディレクトリ操作を伴うワークロードで遅い

## 主要仮説

1. ロック競合で並列度が頭打ちになっている
2. ディレクトリエントリ探索/更新の計算量が大きい
3. ジャーナル同期/メタ更新が高頻度で待ちを生んでいる
4. ハッシュ重複除去の同期実行が write レイテンシを押し上げている

## アイディア候補

### A. 2 段階書込み（同期パス短縮）

- 目的: `write(2)` での待ち時間を短縮
- 発想:
  - 第 1 段階: まず物理ブロックへ直接書込み（短いクリティカルパス）
  - 第 2 段階: ハッシュ計算/重複除去をバックグラウンドで遅延適用
- 利点:
  - フロントの write 応答が速くなる
  - 重複除去計算を CPU 空き時間へ逃がせる
- 注意点:
  - 再マップ時の整合性（クラッシュ時、再起動時）
  - 同一ブロックの同時更新（世代番号/バージョン管理が必要）
  - 読み出しで「直接ブロック」と「重複除去済み参照」の両方を扱う必要

### B. ジャーナル強化（耐久とスループットの両立）

- 目的: メタ整合性を維持しつつ同期コストを抑える
- 発想:
  - group commit 窓の適応制御（固定値から動的へ）
  - メタ優先 flush とデータ遅延 flush の分離
  - リプレイ可能な操作を増やして、復旧を堅牢化
- 利点:
  - 高頻度小書込みで fsync 負荷を平準化
  - 障害復旧の確実性を上げられる
- 注意点:
  - 書込み遅延と耐久保証のトレードオフ設計が必要

### C. ロック範囲改善（並列度向上）

- 目的: スレッド増加に比例して性能が伸びる状態に近づける
- 発想:
  - クリティカルセクションを最小化
  - ディレクトリ操作の read/write ロック分離
  - ハッシュ/bitmap/inode ロック順序の見直しとホットロック分割
- 利点:
  - 競合時の待ち時間短縮
  - CPU 利用率が同じでも有効スループットが上がる可能性
- 注意点:
  - ロック順序変更時のデッドロック検証が必要

### D. ディレクトリ操作特化の改善（追加案）

- 目的: 「mkdir/create/unlink/rename が重い」症状に直接効かせる
- 発想:
  - ディレクトリエントリの探索高速化（線形探索から索引化）
  - 小さな dirent 変更のバッチ更新
  - ネガティブ lookup キャッシュ（存在しない名前の再探索抑制）
- 利点:
  - Git 系/小ファイル大量ワークロードで改善期待が大きい

### E. 観測基盤を先に強化（追加案）

- 目的: 体感ではなく、数字で優先順位を決める
- 発想:
  - lock wait 時間、dirent 探索時間、ジャーナル flush 回数を常時観測
  - 変更前後を同一 workload で比較可能にする

## 実装優先順位（提案）

1. 観測基盤（E）
2. ロック範囲改善（C）
3. ディレクトリ特化改善（D）
4. ジャーナル強化（B）
5. 2 段階書込み（A）

理由:

- 先に計測しないと、A/B/C のどこが真因かを誤る可能性が高い
- A（2 段階）は効果が大きい反面、整合性設計の難度が最も高い
